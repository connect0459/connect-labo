# テストサイズ戦略

## テストサイズの定義

Google Testing Blog 発の Small / Medium / Large による分類を共通語彙として使う。
「ユニットテスト」「インテグレーションテスト」「E2Eテスト」はチームによって指す範囲がブレやすいが、テストサイズは「何に依存するか」で分類するため認識がブレにくい。

参考: <https://testing.googleblog.com/2010/12/test-sizes.html>

| サイズ | 依存範囲 | 実行速度 | 代表例 |
| --- | --- | --- | --- |
| **Small** | 単一プロセス内のみ。外部依存なし | 高速（ms単位） | ユニットテスト |
| **Medium** | ローカルホスト内の複数サービス。DB・キャッシュを含む | 中速（秒単位） | リポジトリ統合テスト |
| **Large** | ネットワーク通信、ブラウザ操作、外部サービス連携を含む | 低速（分単位） | E2E・受け入れテスト |

## エージェントへのテストサイズ指定

テストの「種類」ではなく「サイズ＝制約条件」で伝えることでプロンプトの曖昧さが減る。

```text
# Bad（曖昧）
受け入れテストを書いて

# Good（明確）
Largeテストを書いて。ブラウザ経由でユーザーの購入フローを検証する。外部決済APIはモックしてよい
```

制約条件（何に依存するか・何をモックしてよいか）を明示すると、実現可能性の高いコードが得られる。

## リポジトリへの組み込み指針

各リポジトリの CLAUDE.md / AGENTS.md に以下を明記することを推奨する。

- このリポジトリにおける Small / Medium / Large テストの定義
- Largeテストを書くべき場面（購入フロー・認証フローなど主要ユーザーパス）
- 外部サービスのモック方針（何をモックしてよいか）
- Largeテストの実行タイミング（CI全実行 or 夜間バッチ等）
